## 2. GitHub OAuth Setup

### Step 1: Create GitHub OAuth App

1. Go to **https://github.com/settings/developers**
2. Click **"OAuth Apps"** → **"New OAuth App"**
3. Fill in:

| Field | Local Development | Production |
|-------|------------------|------------|
| **Application name** | `Oply Dev` | `Oply` |
| **Homepage URL** | `http://localhost:3000` | `https://yourdomain.com` |
| **Authorization callback URL** | `http://localhost:3000/api/auth/callback/github` | `https://yourdomain.com/api/auth/callback/github` |

4. Click **"Register application"**
5. Copy the **Client ID** → paste as `GITHUB_CLIENT_ID`
6. Click **"Generate a new client secret"** → paste as `GITHUB_CLIENT_SECRET`

### Callback URL (Critical!)

```
Local:      http://localhost:3000/api/auth/callback/github
Production: https://yourdomain.com/api/auth/callback/github
```

> ⚠️ This URL is handled by NextAuth automatically at `/api/auth/[...nextauth]/route.ts`. You do NOT need to create this route — it's generated by the `handlers` export from `lib/auth.ts`.

### Required Scopes

Oply requests these scopes by default:
- `read:user` — Read user profile
- `user:email` — Read email address

## 3. Google OAuth Setup

### Step 1: Create Google OAuth Credentials

1. Go to **https://console.cloud.google.com/apis/credentials**
2. Click **"Create Credentials"** → **"OAuth client ID"**
3. Select **"Web application"**
4. Fill in:

| Field | Value |
|-------|-------|
| **Name** | `Oply` |
| **Authorized JavaScript origins** | `http://localhost:3000` |
| **Authorized redirect URIs** | `http://localhost:3000/api/auth/callback/google` |

5. Click **"Create"**
6. Copy **Client ID** → paste as `GOOGLE_CLIENT_ID`
7. Copy **Client Secret** → paste as `GOOGLE_CLIENT_SECRET`

### Production Redirect URIs

Add these in Google Console for production:

```
Authorized JavaScript origins:
  https://yourdomain.com

Authorized redirect URIs:
  https://yourdomain.com/api/auth/callback/google
```

### Callback URL (Critical!)

```
Local:      http://localhost:3000/api/auth/callback/google
Production: https://yourdomain.com/api/auth/callback/google
```

### Prerequisites

- Enable the **"Google+ API"** or **"People API"** in Google Cloud Console
- Configure the **OAuth consent screen** (External, publish status: Testing or Production)

## 4. NextAuth Configuration

### How It Works

NextAuth v5 is configured in `app/lib/auth.ts`:

```typescript
// lib/auth.ts
export const { handlers, auth, signIn, signOut } = NextAuth({
  adapter: DrizzleAdapter(db, { ... }),
  providers: [GitHub({ ... }), Google({ ... })],
  pages: { signIn: "/login" },
  callbacks: {
    session: ({ session, user }) => ({
      ...session,
      user: { ...session.user, id: user.id },
    }),
  },
});
```

### Auth Routes (Auto-Generated by NextAuth)

| Route | Method | Purpose |
|-------|--------|---------|
| `/api/auth/signin` | GET | Shows sign-in page (redirects to `/login` via `pages` config) |
| `/api/auth/signout` | POST | Signs out the user |
| `/api/auth/callback/github` | GET | GitHub OAuth callback handler |
| `/api/auth/callback/google` | GET | Google OAuth callback handler |
| `/api/auth/session` | GET | Returns current session JSON |
| `/api/auth/csrf` | GET | Returns CSRF token |
| `/api/auth/providers` | GET | Lists available providers |

### Using Auth in Server Components

```typescript
import { auth } from "@/lib/auth";

export default async function Page() {
  const session = await auth();
  if (!session?.user) redirect("/login");
  // session.user.id, session.user.email, session.user.image
}
```

### Using Auth in Client Components

```typescript
"use client";
import { signIn, signOut } from "next-auth/react";

// Sign in with GitHub
signIn("github", { callbackUrl: "/dashboard" });

// Sign in with Google
signIn("google", { callbackUrl: "/dashboard" });

// Sign out
signOut({ callbackUrl: "/" });
```

### Protected API Routes

```typescript
import { auth } from "@/lib/auth";

export async function GET() {
  const session = await auth();
  if (!session?.user) {
    return NextResponse.json({ error: "Unauthorized" }, { status: 401 });
  }
  // session.user.id is available
}
```

### NEXTAUTH_SECRET

Generate a secure random secret:

```bash
openssl rand -base64 32
```

Paste the output as `NEXTAUTH_SECRET` in `.env`.

### NEXTAUTH_URL

| Environment | Value |
|-------------|-------|
| Local dev | `http://localhost:3000` |
| Production | `https://yourdomain.com` |

> On Vercel, this is set automatically via `VERCEL_URL`.

## 5. Stripe Billing Setup

### Step 1: Create Stripe Account

1. Go to **https://dashboard.stripe.com/register**
2. After signup, navigate to **Developers** → **API keys**
3. Copy:
   - **Publishable key** (`pk_test_...`) → `NEXT_PUBLIC_STRIPE_PUBLIC_KEY`
   - **Secret key** (`sk_test_...`) → `STRIPE_SECRET_KEY`

### Step 2: Create Products & Prices

Go to **https://dashboard.stripe.com/products** → **"Add product"**

Create 3 products:

| Product | Price | Billing |
|---------|-------|---------|
| **Oply Starter** | $19/month | Recurring |
| **Oply Pro** | $49/month | Recurring |
| **Oply Enterprise** | $199/month | Recurring |

After creating each product, copy the **Price ID** (starts with `price_`):

```env
STRIPE_STARTER_PRICE_ID=price_1ABC...
STRIPE_PRO_PRICE_ID=price_1DEF...
STRIPE_ENTERPRISE_PRICE_ID=price_1GHI...
```

### Step 3: Set Up Webhook

#### Local Development (Stripe CLI)

1. Install Stripe CLI: https://stripe.com/docs/stripe-cli
2. Login:
   ```bash
   stripe login
   ```
3. Forward webhooks to your local server:
   ```bash
   stripe listen --forward-to http://localhost:3000/api/v1/billing/webhook
   ```
4. Copy the **webhook signing secret** (starts with `whsec_`) → `STRIPE_WEBHOOK_SECRET`

#### Production

1. Go to **https://dashboard.stripe.com/webhooks**
2. Click **"Add endpoint"**
3. Fill in:

| Field | Value |
|-------|-------|
| **Endpoint URL** | `https://yourdomain.com/api/v1/billing/webhook` |
| **Events to send** | Select the following: |

**Required Webhook Events:**

| Event | Purpose |
|-------|---------|
| `checkout.session.completed` | User completed payment → create subscription |
| `invoice.paid` | Recurring payment succeeded → renew subscription |
| `customer.subscription.updated` | Plan changed / canceled at period end |
| `customer.subscription.deleted` | Subscription fully canceled |

4. Copy the **Signing secret** → `STRIPE_WEBHOOK_SECRET`

### Billing API Endpoints

| Method | Endpoint | Auth | Purpose |
|--------|----------|------|---------|
| POST | `/api/v1/billing/checkout` | Session | Creates Stripe Checkout session, returns redirect URL |
| POST | `/api/v1/billing/portal` | Session | Creates Stripe Billing Portal session, returns redirect URL |
| POST | `/api/v1/billing/webhook` | Stripe Sig | Handles Stripe webhook events (no user auth needed) |
| GET | `/api/v1/billing/usage` | Session | Returns current usage stats for the logged-in user |

### Checkout Flow

```
1. User clicks "Upgrade to Pro" on /dashboard/billing
2. Frontend calls POST /api/v1/billing/checkout { plan: "pro" }
3. Backend creates Stripe Checkout Session
4. Backend returns { url: "https://checkout.stripe.com/..." }
5. Frontend redirects to Stripe Checkout
6. User pays
7. Stripe fires webhook: checkout.session.completed
8. Backend creates subscription record in DB
9. Stripe redirects user to /dashboard/billing?success=true
```

### Billing Portal Flow

```
1. User clicks "Manage via Stripe" on /dashboard/billing
2. Frontend calls POST /api/v1/billing/portal
3. Backend creates Stripe Billing Portal session
4. Backend returns { url: "https://billing.stripe.com/..." }
5. Frontend redirects to Stripe Portal
6. User can: update payment, cancel, view invoices
7. Stripe Portal redirects back to /dashboard/billing
```

## 6. GitHub Webhooks Setup

### Webhook Endpoint

```
Local:      http://localhost:3000/api/v1/webhooks/github  (use ngrok or smee.io)
Production: https://yourdomain.com/api/v1/webhooks/github
```

### Setting Up in GitHub

1. Go to your repository → **Settings** → **Webhooks** → **"Add webhook"**
2. Fill in:

| Field | Value |
|-------|-------|
| **Payload URL** | `https://yourdomain.com/api/v1/webhooks/github` |
| **Content type** | `application/json` |
| **Secret** | Same value as `GITHUB_WEBHOOK_SECRET` in your `.env` |

3. Select events:
   - `push` — Trigger pipeline on push
   - `pull_request` — Trigger pipeline on PR

### Using ngrok for Local Development

```bash
# Install ngrok
npm install -g ngrok

# Start tunnel
ngrok http 3000

# Use the HTTPS URL as your webhook endpoint:
# https://abc123.ngrok.io/api/v1/webhooks/github
```

### Alternative: smee.io

```bash
# Go to https://smee.io and click "Start a new channel"
# Install smee client
npm install -g smee-client

# Forward events
smee -u https://smee.io/your-channel-id -t http://localhost:3000/api/v1/webhooks/github
```

## 7. CLI Authentication Flow

### Device Code Flow (like GitHub CLI / Vercel CLI)

```
┌──────────────┐                    ┌──────────────┐                   ┌──────────┐
│   CLI (oply)  │                    │  Oply Backend │                   │  Browser  │
└──────┬───────┘                    └──────┬───────┘                   └─────┬────┘
       │                                    │                                 │
       │  POST /api/v1/cli/login/start      │                                 │
       │───────────────────────────────────>│                                 │
       │                                    │                                 │
       │  { code: "ABCD-1234",             │                                 │
       │    verifyUrl: ".../cli-auth" }     │                                 │
       │<───────────────────────────────────│                                 │
       │                                    │                                 │
       │  Print: "Go to URL, enter code"    │                                 │
       │  Opens browser automatically       │─────────────────────────────── >│
       │                                    │                                 │
       │                                    │    User logs in (GitHub/Google) │
       │                                    │< ────────────────────────────── │
       │                                    │                                 │
       │                                    │    User enters code "ABCD-1234" │
       │                                    │< ────────────────────────────── │
       │                                    │                                 │
       │                                    │  POST /api/v1/cli/login/verify  │
       │                                    │  { code: "ABCD-1234" }          │
       │                                    │  → Generates token              │
       │                                    │  → Links user to CLI session    │
       │                                    │                                 │
       │  POST /api/v1/cli/login/poll       │                                 │
       │  { code: "ABCD-1234" }             │                                 │
       │───────────────────────────────────>│                                 │
       │                                    │                                 │
       │  { status: "verified",             │                                 │
       │    token: "oply_abc123...",         │                                 │
       │    user: { email, name } }         │                                 │
       │<───────────────────────────────────│                                 │
       │                                    │                                 │
       │  Token saved to ~/.oply/config.json │                                │
       └────────────────────────────────────┘                                 │
```

### CLI Login API Endpoints

| Method | Endpoint | Auth Required | Purpose |
|--------|----------|---------------|---------|
| POST | `/api/v1/cli/login/start` | ❌ No | Generates device code + returns verify URL |
| POST | `/api/v1/cli/login/verify` | ✅ Session | Authenticated user submits code to link CLI |
| POST | `/api/v1/cli/login/poll` | ❌ No | CLI polls until code is verified |

### Request/Response Examples

#### Start Login

```bash
curl -X POST http://localhost:3000/api/v1/cli/login/start \
  -H "Content-Type: application/json" \
  -d '{"userAgent": "oply-cli/1.0.0"}'
```

Response:
```json
{
  "code": "ABCD-1234",
  "expiresAt": "2026-04-12T20:00:00.000Z",
  "verifyUrl": "http://localhost:3000/cli-auth"
}
```

#### Poll Login (CLI repeats every 3s)

```bash
curl -X POST http://localhost:3000/api/v1/cli/login/poll \
  -H "Content-Type: application/json" \
  -d '{"code": "ABCD-1234"}'
```

Response (pending):
```json
{ "status": "pending" }
```

Response (verified):
```json
{
  "status": "verified",
  "token": "oply_a1b2c3d4e5f6...",
  "user": {
    "id": "uuid-here",
    "email": "user@example.com",
    "name": "John Doe",
    "image": "https://avatars.githubusercontent.com/..."
  }
}
```

### Token Storage

The CLI stores the token at:

```
~/.oply/config.json
```

Contents:
```json
{
  "token": "oply_a1b2c3d4e5f6...",
  "user": {
    "id": "uuid",
    "email": "user@example.com",
    "name": "John Doe"
  },
  "authenticatedAt": "2026-04-12T19:45:00.000Z"
}
```

### CLI Commands

```bash
# Login (starts device code flow)
oply login

# Check login status
oply login --status

# Logout (deletes stored token)
oply login --logout
```

## 9. Database Setup

### Option A: Neon PostgreSQL (recommended)

1. Go to **https://neon.tech** → Create project
2. Copy the connection string → paste as `DATABASE_URL`

### Option B: Supabase PostgreSQL

1. Go to **https://supabase.com** → Create project
2. Go to Settings → Database → Connection string (URI)
3. Copy → paste as `DATABASE_URL`

### Option C: Local PostgreSQL

```bash
# Using Docker
docker run -d \
  --name oply-postgres \
  -e POSTGRES_USER=oply \
  -e POSTGRES_PASSWORD=oply \
  -e POSTGRES_DB=oply \
  -p 5432:5432 \
  postgres:16

# Connection string
DATABASE_URL=postgresql://oply:oply@localhost:5432/oply
```

### Push Schema to Database

```bash
cd app
npm run db:push
```

This creates all tables:
- `user`, `account`, `session`, `verificationToken` (NextAuth)
- `organization`, `project`, `repository` (Core)
- `environment`, `workflow`, `pipeline_run`, `task_run` (Pipeline engine)
- `deployment` (Deployments)
- `ai_conversation`, `ai_message`, `ai_automation`, `ai_node` (AI)
- `cli_session` (CLI auth)
- `subscription` (Billing)
- `usage` (Usage tracking)

### View Database (Drizzle Studio)

```bash
cd app
npm run db:studio
```

Opens at: **https://local.drizzle.studio**

## 11. Production Deployment

### Vercel (Recommended)

1. Push to GitHub
2. Import in Vercel: https://vercel.com/import
3. Set environment variables in Vercel dashboard
4. Update all callback URLs:

```
NEXTAUTH_URL=https://yourdomain.com

GitHub OAuth callback:  https://yourdomain.com/api/auth/callback/github
Google OAuth callback:  https://yourdomain.com/api/auth/callback/google
Stripe Webhook URL:     https://yourdomain.com/api/v1/billing/webhook
GitHub Webhook URL:     https://yourdomain.com/api/v1/webhooks/github
```

### Checklist Before Going Live

- [ ] `NEXTAUTH_URL` set to production domain
- [ ] `NEXTAUTH_SECRET` is a secure random string
- [ ] GitHub OAuth callback URL updated to production domain
- [ ] Google OAuth callback URL updated to production domain
- [ ] Google OAuth consent screen published (not "Testing" mode)
- [ ] Stripe webhook endpoint created for production domain
- [ ] `STRIPE_WEBHOOK_SECRET` is the production webhook secret
- [ ] Stripe products/prices created and price IDs set
- [ ] `DATABASE_URL` points to production PostgreSQL
- [ ] `npm run db:push` run against production database
- [ ] `GROQ_API_KEY` is valid and has sufficient quota
